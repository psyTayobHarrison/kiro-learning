# Steering Comparison: Steered (spec-req-first) vs Unsteered (vibe)

## Task

> "Add a new feature to the expense tracker: support filtering expenses by a date range instead of just a single category. New endpoint: GET /expenses/range?start=YYYY-MM-DD&end=YYYY-MM-DD. Also add a 'notes' field to expenses — optional, free text, separate from the existing 'description' field. Make whatever changes are needed to the entity, repository, and database to support this."

## Summary

The unsteered (vibe) version would require significantly more manual correction. Its most critical deficiency is the in-memory aggregation pattern in `getSummary()` — this virtually guarantees a date-range endpoint would also be implemented as "fetch all, filter in Java" rather than a proper repository query. Additionally, it lacks `precision`/`scale` on monetary columns, uses `LocalDateTime` instead of `Instant`, has no test profile, and provides no structural guardrails to ensure a `notes` field gets correct column mapping. I'd estimate **3–4 follow-up prompts** to bring the unsteered version in line: one to fix the summary to use JPQL, one to add a proper derived/query method for the date-range endpoint, one to fix the entity conventions (precision/scale, Instant, explicit column annotations), and possibly one more to set up the test profile with H2.
